# 空气堡电商数据分析平台项目总控文档

## 1. 项目定位

本项目是一个面向天猫、京东、抖音等电商平台的本地数据分析平台，核心用途是帮助空气堡把平台后台导出的 Excel / CSV 表格转化为可读、可算、可判断的经营数据。

当前项目不追求一步到位做成 SaaS，也不追求自动采集。第一阶段只解决一条主链路：

```text
手动上传表格 -> 字段识别 -> 数据清洗 -> 指标计算 -> 运营判断
```

平台要服务的经营问题包括：

- 哪些商品贡献销售额？
- 哪些商品访客高但转化低？
- 哪些商品收藏、加购有潜力但支付不足？
- 哪些商品退款、异常、数据缺失需要优先检查？
- 哪些运营动作后续值得被 AI 报告进一步解释？

## 2. 第一阶段目标

第一阶段目标是做出一个本地可稳定运行的基础版本，支持用户手动上传天猫、京东、抖音等平台导出的 Excel / CSV 表格，并完成基础分析。

第一阶段必须跑通：

1. 用户手动上传 Excel / CSV 文件。
2. 系统读取表格表头和数据行。
3. 系统识别或提示匹配核心字段。
4. 系统对金额、人数、订单数、比例、日期等字段做基础清洗。
5. 系统计算核心经营指标。
6. 系统展示数据表格、统计卡片、商品排行和异常商品。
7. 系统能输出初步运营判断，但不把它包装成最终经营结论。

## 3. 当前阶段边界

当前阶段只做手动上传 Excel / CSV。

允许做：

- 本地项目初始化。
- 手动选择文件并解析。
- Excel / CSV 表头识别。
- 字段映射和字段字典维护。
- 基础数据清洗。
- 核心指标计算。
- 商品排行和异常商品识别。
- AI 报告入口预留。

不允许做：

- 不接天猫 API。
- 不接京东 API。
- 不接抖音 API。
- 不做爬虫。
- 不做自动登录。
- 不做自动抓取平台后台。
- 不做 SaaS 付费系统。
- 不做复杂权限系统。
- 不做多租户后台。
- 不做定时任务采集。
- 不把字段含义不明的数据包装成确定结论。

## 4. 推荐技术路线

第一阶段推荐技术路线：

- 前端框架：Next.js + React + TypeScript。
- 样式方案：Tailwind CSS。
- Excel 解析：xlsx。
- CSV 解析：Papa Parse。
- 表格展示：TanStack Table。
- 图表展示：Recharts。
- 本地开发阶段：优先使用浏览器内存和前端解析。
- 后续需要保存历史记录时：再引入 SQLite + Prisma。

技术路线原则：

- 先稳定解析真实表格，再做漂亮页面。
- 先统一字段口径，再做复杂指标。
- 先本地可用，再考虑 SaaS 化。
- 先人工上传，再考虑自动化。
- 先让数据支持经营判断，再考虑 AI 报告。

## 5. 第一阶段核心模块

### 5.1 文件上传

用户手动上传一个或多个 Excel / CSV 文件。系统只负责读取用户选择的本地文件，不自动访问平台后台。

### 5.2 表格解析

系统解析文件中的 sheet、表头、数据行、空行、合计行和异常格式。

### 5.3 字段识别

系统根据字段字典识别平台导出表中的字段，例如访客数、支付金额、支付买家数、转化率、收藏数、加购数等。

### 5.4 数据清洗

系统统一处理：

- 金额字段。
- 整数字段。
- 小数字段。
- 百分比字段。
- 日期字段。
- 空值。
- 合计行。
- 重复行。

### 5.5 指标计算

系统基于清洗后的数据计算：

- 销售额。
- 访客数。
- 浏览量。
- 支付买家数。
- 支付订单数。
- 支付转化率。
- 客单价。
- 收藏数。
- 加购数。
- 退款金额。
- 退款订单数。

### 5.6 运营判断

系统第一阶段只做基础判断，不做复杂结论：

- 高访客低转化商品。
- 高加购低支付商品。
- 高销售额商品。
- 退款异常商品。
- 缺失关键字段的数据。
- 数据质量风险提示。

## 6. 禁止事项

这些事项在第一阶段禁止进入开发范围：

1. 不接天猫 API。
2. 不接京东 API。
3. 不接抖音 API。
4. 不做爬虫。
5. 不做自动登录。
6. 不做自动采集。
7. 不做平台账号授权。
8. 不做 SaaS 付费系统。
9. 不做复杂权限。
10. 不做多租户。
11. 不做自动投放建议。
12. 不把 AI 报告作为第一阶段核心功能。
13. 不做无法解释经营用途的装饰性大屏。

## 7. 成功标准

第一阶段完成时，平台至少能回答：

- 这份表格来自哪个平台？
- 系统识别出了哪些字段？
- 哪些字段缺失或含义不确定？
- 销售额、访客数、买家数、转化率、客单价分别是多少？
- 哪些商品销售额最高？
- 哪些商品访客高但支付转化低？
- 哪些商品收藏或加购不错但支付不足？
- 哪些数据因为字段缺失、格式异常或口径不明，暂时不能用于判断？

## 8. 数据证据原则

本项目不把界面成功、文件上传成功、图表显示成功等同于经营分析成功。

每个分析结果都必须能追溯到：

- 原始上传文件。
- 原始表头。
- 字段映射规则。
- 清洗规则。
- 计算公式。
- 判断条件。

如果字段含义无法确认，系统必须标记为未知或待确认，不能强行归类。

## 9. 后续扩展方向

第一阶段稳定后，再考虑：

- 历史上传记录。
- 多文件合并分析。
- 店铺维度分析。
- 时间趋势分析。
- 活动前后对比。
- AI 运营诊断报告。
- 多用户权限。
- SaaS 后台。
- 平台 API 或官方授权接口。

这些扩展都必须建立在第一阶段手动上传链路稳定的基础上。

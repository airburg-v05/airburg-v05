1. 四份真实报表均已找到，实际源目录为 `/Users/zongji/Desktop/平台报表/天猫`，不是最初预期的 `/Users/zongji/Desktop/平台报表-天猫`。

4. CSV 两份文件编码均识别为 `GB18030`，后续解析不能只按 UTF-8。

结论：商品报表的计划均可在计划报表中找到；计划报表多出 1 个商品报表未覆盖计划，可能是无商品明细、无消耗或非商品主体计划，需要后续用计划状态/场景确认。

## 13. 推荐四表处理顺序

1. 复制到 `private-samples`，确认 `.gitignore` 生效。
2. 读取表格，只保留表头和必要字段，不输出明细。
3. 所有 ID 字段转字符串。
4. 生意参谋商品表生成 `fact_product_daily`。
5. 商品报表先按 `日期 + 主体ID` 汇总，再关联商品表。
6. 计划报表按 `日期 + 计划ID` 汇总，供计划看板使用。
7. 售后表先删除敏感字段，再按商品和日期聚合。
8. 用聚合后的售后数据计算 GSV、退款率和售后风险。
9. 形成脱敏汇总包设计记录，仅作为未来 AI/千问模块恢复时的参考；当前阶段不接入、不调用、不开发 AI。

## 15. 当前缺失字段与口径歧义

字段层面四类指标均可实现，但存在以下口径需要人工确认：

1. GSV 使用生意参谋 `成功退款金额`，还是售后 `退款总额` / `退给买家金额`。
2. 退款金额使用申请金额、成功金额，还是退给买家金额。
3. 推广成交金额使用 `总成交金额`，是否包含预售需确认。
4. 计划报表是 7 天周期，和单日商品/售后表合并时需按日期对齐。
5. 去退推广费比的退款归因：按商品付款日、退款申请日，还是退款完结日。
6. 平均访问页面数若做汇总，需要确认原始分子字段；不能直接平均。
7. 售后待处理状态集合需要根据 `退款状态` 枚举定义。
8. 超时未处理的当前时间口径需要确认：实时当前时间、报表导出时间，还是分析日期结束时间。
9. `退款原因标签`为空，原因分布应先使用 `买家退款原因`。
10. 当前 localStorage 存 `previewRows`，未来售后接入前必须改为脱敏聚合数据。

## Data Import Page

`/upload` is the only data import entry for V0.5B.

The page supports:

1. Platform selection, with Tmall enabled and JD, PDD, Douyin, and Youzan disabled.
2. Store selection and adding a new Tmall store.
3. Multiple file selection through one file input.
4. Automatic source recognition for the four Tmall reports.
5. One import action that runs parse, validate, aggregate, prepare, readback, activate, and save.
6. Reimport context from `/upload/quality` without auto-submitting.

The page does not contain the old four-card upload layout and does not contain the old "开始四源分析" flow.

## Import History Page

`/upload/history` is read-only.

The page supports:

1. Platform, store, status, date, and search filters.
2. Deduplicated import history rows.
3. Active and inactive dataset state display.
4. A details drawer with safe source counts and activation timeline.
5. A link to view the current batch quality.
6. Escape-to-close behavior and focus return.

The history page must not expose delete, rollback, overwrite, or reimport actions.

## Data Quality Page

`/upload/quality` is the repair guidance center.

The page supports:

1. Platform, store, batch, issue type, status, and search filters.
2. Safe issue categories and counts.
3. Repair suggestions.
4. A safe reimport entry that returns to `/upload` with platform, store, and source batch context.

The quality page does not show raw warning text, file name history, raw rows, preview rows, or sensitive after-sales values.

## Multi-Store Isolation

The V0.5B data center supports multiple stores under the Tmall platform.

Frozen rules:

1. Every import must bind to `platformCode` and `storeId`.
2. The same `productId` can exist in multiple stores without merging facts.
3. The same `planId` can exist in multiple stores without merging facts.
4. The default Tmall store can write legacy compatibility data.
5. Non-default stores must not overwrite legacy compatibility keys.

## Duplicate And Conflict

Frozen rules:

1. Reimporting the same file fingerprints into the same store returns `already_imported`.
2. Duplicate import must not increase facts, batches, or activation journal entries.
3. Same-key, different-content conflicts return `conflict`.
4. Conflict must not change the active pointer.
5. Conflict must not overwrite legacy compatibility data.
6. There is no forced overwrite flow in V0.5B.

## Safe Reimport

Safe reimport only starts from `/upload/quality`.

Frozen rules:

1. Reimport returns to `/upload`.
2. Platform and store are preselected from the source batch context.
3. The page must not auto-submit.
4. A corrected import creates a new batch.
5. The original batch remains read-only.

## IndexedDB Activation

V0.5B uses the V2 IndexedDB persistence engine created in V0.5A.

Frozen rules:

1. Prepare, readback, and activate must all pass before a dataset becomes active.
2. Active pointer changes are atomic.
3. Failed or conflict import attempts do not change the active pointer.
4. The production database name remains `airburg-v05`.
5. V0.5B runtime audits use isolated audit databases and must not touch production data.

## Legacy Compatibility

The default Tmall store continues to write `airburg_tmall_analysis_v2` for legacy dashboards.

Frozen rules:

1. Default-store imports save legacy compatibility data.
2. Non-default store imports do not write the legacy key.
3. Legacy dashboards must continue reading default-store compatible data.
4. Second-store data must not enter legacy dashboards.

## Privacy Boundary

V0.5B only stores and displays privacy-safe after-sales aggregates.

Forbidden in pages, console output, IndexedDB safe records, and validation output:

1. Raw rows.
2. Preview rows.
3. Raw file contents.
4. File name history outside the current upload file list.
5. Raw warning text.
6. Order identifiers.
7. Refund identifiers.
8. Transaction identifiers.
9. Phone numbers.
10. Addresses.
11. Logistics information.
12. Buyer explanations.
13. Seller notes.
14. Operators and subaccounts.
